# 缓存管理界面优化设计

## 背景

当前工作区顶部有“缓存列表”和“请求管理”两个入口。右侧页面实际承担的是缓存条目的筛选、选择、响应体预览与编辑，因此“请求管理”容易让人误以为是在管理原始请求或代理请求流。该入口更适合命名为“缓存管理”。

当前界面功能已经完整，但视觉层级偏旧：列表、详情、编辑器、操作按钮之间缺少明确的工作台结构；顶部过滤和详情信息占用空间较散；JSON 编辑器与页面自身控件之间风格割裂。

## 目标

将“请求管理”升级为更现代的“缓存管理”工作台，使用户能更清晰地完成以下流程：

1. 搜索和筛选缓存条目。
2. 在左侧列表快速定位缓存。
3. 在右侧查看缓存元信息、响应状态和响应体。
4. 在 JSON 编辑器中预览、编辑、保存或恢复响应体。

## 命名

- 顶部 Tab 文案从“请求管理”改为“缓存管理”。
- 相关无障碍标签和按钮文案同步使用“缓存管理”。
- 缓存列表中“在请求管理中打开”改为“在缓存管理中打开”。
- 代码中的 DOM id 和状态字段暂不重命名，避免扩大改动范围；本次只调整用户可见文案和样式。

## 视觉方向

采用用户选择的 B 方案：现代工作台式布局。

核心特征：

- 页面标题区域突出“缓存管理”，副文案说明“查看、筛选、编辑缓存响应”。
- “缓存列表 / 缓存管理”以更轻量的分段控件呈现。
- 管理视图使用左侧列表、右侧详情和编辑器的清晰分区。
- 左侧列表增加结果数量和当前选择提示，选中态更稳重。
- 右侧详情信息放入轻量信息条，避免散落在页面大块空白中。
- 编辑器外层形成独立工作区，预览/编辑/原始和保存类操作放在同一条工具栏内。

## 范围

本次只优化已有静态 UI 和前端渲染：

- 修改 `public/index.html` 中的可见文案和少量结构。
- 修改 `public/styles.css` 中的管理页布局、列表、详情、编辑器区域样式。
- 修改 `public/app.js` 中必要的渲染文案，例如列表结果统计和“在缓存管理中打开”。
- 更新现有 UI 静态测试，覆盖新文案和关键样式钩子。

不改动：

- 缓存读写 API。
- JSON 编辑器库和右键复制能力。
- 数据存储、回放、TTL、导入导出逻辑。
- 代码内部命名的大规模重构。

## 验收标准

- 用户可见的入口名称为“缓存管理”。
- 管理页在桌面宽屏下呈现工作台式布局：左侧列表、右侧详情、下方编辑器层级清晰。
- 页面仍能在窄屏下堆叠显示，不出现明显重叠或文本溢出。
- JSON 编辑器区域高度稳定，和页面控件视觉上更协调。
- 现有缓存管理功能不回退：筛选、上一条/下一条、预览/编辑/原始、格式化、保存、恢复仍可用。
- `rtk npm test` 和 `rtk npm run build` 通过。

## 测试策略

- 增加关键样式类或结构断言，保证工作台布局不会被意外移除。
- 运行完整测试和构建，确认前端静态资源和 TypeScript 构建路径正常。
